# M4.5 — MCP Surface

**Goal:** Stand up the MCP server as a co-equal product surface. Agents authenticate with capability tokens (M1) and call the same Convex actions the UI uses (M2/M3/M4) — no domain logic in the MCP layer. Every agent action lands in the shared household state with `actor_kind='agent'` attribution.

**Exit criterion:**
1. `just mcp-dev` boots the MCP server in stdio. Claude Desktop (or any MCP client) configured with a seeded agent token can list tools and successfully call `capture_item`, `query_items`, `log_item`.
2. Scripted MCP client test (in `e2e/mcp/`) runs the M6 agent leg green.
3. CI parity check: every Convex `mutation` exported from `packages/convex` that is intended for agent use has a matching tool registered in `apps/mcp` (allowlist file lives in `packages/shared/src/tools/registry.ts`).
4. A revoked or expired token is rejected with a 401-equivalent MCP error and no state change.

## Tasks

### Tool contracts (`packages/shared/src/tools/`)
- [ ] `registry.ts` — declarative list of tool definitions: `name`, `description` (LLM-facing), Zod input schema, Zod output schema, the Convex action it wires to, required scope (`read` | `write` | `archive`)
- [ ] Tools in MVP scope:
  - `capture_item` — title-only quick capture, returns item id
  - `query_items` — filter by status, owner, area, waiting_for, recently-updated; bounded result size
  - `get_item` — by id, includes children counts and most recent N logs
  - `log_item` — append a log entry to an item; `tool_name` is auto-set to `log_item`
  - `transition_item` — wraps `setItemStatus`; rejects illegal transitions via the same shared state machine
  - `assign_item` — wraps `assignItem`
  - `update_item` — wraps `updateItem` (excludes status, same as UI)
  - `search` — wraps the M5 search query (gated; returns empty until M5 lands)
- [ ] Each tool description is written for the agent reader: explains preconditions, gives an example input, names the failure modes (illegal transition, missing waiting_for, etc.)
- [ ] Output schemas always include the resulting item shape so agents can chain calls without a follow-up `get_item`

### Convex action wrappers (`packages/convex/agent.ts`)
- [ ] HTTP action `agentCall({ tool, args, token })`:
  - Resolves `token` via `resolveAgentToken` (M1 helper)
  - Rejects if revoked, expired, or `tool` not in `allowed_tools`
  - Validates `args` against the shared Zod input schema for the tool
  - Builds an `Actor = { kind: 'agent', id: agent._id, displayName: agent.display_name }`
  - Dispatches to the corresponding internal mutation/query (the same ones used by the UI), passing `actor`
  - Updates `last_used_at` on the token
  - Returns the validated output, or a structured error
- [ ] Cross-household guard: agent `_id` must belong to the household that owns the target item (re-checked inside each mutation; the agent endpoint is a second line of defense)
- [ ] Rate limit: simple per-token token-bucket (e.g., 60 calls/min) — reject with retriable error

### MCP server (`apps/mcp`)
- [ ] Server constructor reads tool registry from `packages/shared` and registers each as an MCP tool
- [ ] Auth middleware:
  - **stdio transport:** token comes from `GTDVIBE_AGENT_TOKEN` env var (set by the agent's process / Claude Desktop config)
  - **HTTP transport:** token comes from `Authorization: Bearer <token>` header
  - Token is forwarded to `agentCall`; the MCP server itself never trusts client-supplied actor identity
- [ ] Tool handlers are thin: validate via shared Zod, call `agentCall`, format result/error as MCP `CallToolResult`
- [ ] Errors map cleanly: invalid args → MCP `InvalidParams`; auth failure → `Unauthorized`-equivalent; domain rejection → `InternalError` with a structured `{ code, message }` so agents can recover
- [ ] Server logs every call to stderr (tool, token-id-prefix, latency, outcome) — never logs the plaintext token or item bodies

### Parity CI check
- [ ] Script in `packages/convex/scripts/check-tool-parity.ts`:
  - Reads exports from `packages/convex/items.ts`, `packages/convex/logs.ts`, etc.
  - Reads tool registry from `packages/shared/src/tools/registry.ts`
  - Reads an explicit `non-agent-mutations.txt` allowlist for mutations intentionally not exposed (e.g., `deleteItem`)
  - Fails if any non-allowlisted mutation lacks a tool, or any tool references a non-existent mutation
- [ ] Wired into `just check`

### Settings UI updates (`apps/web/app/settings/agents/`)
- [ ] Per-agent: editable allowed-tools checklist (only tools from registry; UI fetches list from a `tools.list()` query)
- [ ] Per-agent: "last used" + last 10 tool calls (tool name, timestamp, outcome) — read from a new lightweight `agent_call_log` view if needed, otherwise from item logs filtered by `actor_id`
- [ ] Big red "Revoke all tokens for this agent" button

### Tests
- [ ] Vitest (`apps/mcp`): tools registry round-trips through the server's `tools/list`
- [ ] Vitest (`packages/convex`): `agentCall` rejects unknown tool, revoked token, expired token, tool-not-in-allowed-set, malformed args, cross-household target
- [ ] Vitest: `agentCall` for `capture_item` writes an item with `created_by_kind='agent'` and an initial `log_type='system'` "Created" log attributed to the agent
- [ ] Vitest: `transition_item` to `waiting` without `waiting_for` rejected via the shared state machine — confirms zero divergence from UI path
- [ ] Vitest: rate limiter rejects the 61st call in a minute, allows the 1st of the next minute
- [ ] Scripted MCP client (`e2e/mcp/agent-leg.spec.ts`) implementing M6 steps A1–A5

### Demo seed updates
- [ ] `just seed` issues `alex-claude` a token with `allowed_tools = ['query_items', 'get_item', 'log_item', 'search']` (read + log only — no transitions, no archives, no captures from the autonomous demo agent)
- [ ] Issues a separate `alex-capture` agent token with `['capture_item']` to demo the capture-via-MCP path without giving the read agent write power
- [ ] Both written to `.secrets/agent-tokens.json`

## Verification
- `just mcp-dev` + Claude Desktop config → Claude can list tools, query items, append a log
- `just mcp-e2e` → green
- Tool parity check fails a sample PR that adds a mutation without a tool, passes once the tool is registered
- Revoking the token in `/settings/agents` immediately causes the next MCP call to fail

## Out of scope
- HTTP transport hosted in production (M7 publishes the URL; auth + rate limit live now, but TLS/edge config is M7)
- OAuth / dynamic client registration — opaque tokens are the MVP
- Agent-initiated multi-step workflows beyond what the listed tools support
- Streaming tool results (MCP supports it; not needed for these operations)
- Per-tool argument-level fine grain ACLs (e.g., "this agent can only `transition_item` to `waiting`") — coarse `allowed_tools` only at MVP
- LLM-side prompt templates / system prompts for the agent — the agent owner provides those
